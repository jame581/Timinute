# v2.3.1 Security Patch — Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Close all 12 open GitHub code-scanning alerts on `jame581/Timinute` without suppressing or dismissing any of them, and add baseline dependency/header hardening.

**Architecture:** Five independent changes to the existing ASP.NET Core server. Three are deletions of dead code (an unreachable Identity page, an unreferenced 2.2 MB static-asset tree), one is a refactor that breaks a taint path by logging database-sourced values instead of raw route parameters, and one adds a small response-header middleware. No new dependencies, no schema change, no user-visible behaviour change.

**Tech Stack:** .NET 10, ASP.NET Core, Blazor WebAssembly (hosted), ASP.NET Core Identity + Razor Pages, Duende IdentityServer (unchanged in this release), xUnit + Moq, `WebApplicationFactory<Program>` for integration tests.

**Spec:** `docs/superpowers/specs/2026-07-14-v2.3.1-security-patch-design.md`

## Global Constraints

- **Nothing is suppressed.** No CodeQL `paths-ignore`, no alert dismissals in the GitHub UI. Every alert is closed by a code change or it stays open and visible.

- **Duende IdentityServer is not touched.** The auth migration is v2.4. Do not modify `IdentitySetup()` in `Server/Program.cs`, and do not touch the `Duende.*` package references.

### Task 2: Sanitize `returnUrl` on GET

Closes alerts **#25** and **#26** (`cs/web/xss` on `LoginWith2fa.cshtml:12,36`).

These two are **false positives for XSS** — Razor HTML-encodes `@Model.ReturnUrl` inside the `asp-route-returnUrl` tag helper, so nothing executes. They are worth fixing anyway because they point at a genuine weakness: `LoginWith2fa.cshtml.cs:82` assigns the raw query-string value to a public property that gets rendered, and the value is only ever validated later, at redirect time, by `LocalRedirect`. Validating on GET closes the open-redirect window and breaks the taint path.

using Microsoft.AspNetCore.Mvc;

namespace Timinute.Server.Helpers
{
    /// <summary>
    /// Normalizes an untrusted <c>returnUrl</c> to a URL that is guaranteed to be local
    /// to this application.
    ///
    /// Identity's scaffolded pages only validate returnUrl at redirect time (LocalRedirect
    /// throws on a foreign URL), but they assign the raw query-string value to a public
    /// property that is rendered into the page before any redirect happens. Sanitizing on
    /// GET closes the open-redirect window and keeps untrusted input out of the markup.
    /// </summary>
    public static class ReturnUrlSanitizer
    {
        public static string Sanitize(IUrlHelper url, string? returnUrl)
        {
            ArgumentNullException.ThrowIfNull(url);

            if (!string.IsNullOrEmpty(returnUrl) && url.IsLocalUrl(returnUrl))
            {
                return returnUrl;
            }

            return url.Content("~/");
        }
    }
}
```

### Task 3: Log database-sourced IDs, not raw route parameters

Closes alerts **#18–#22** (`cs/log-forging`, 5 alerts).

Every flagged log call already uses structured logging, and every one of them sits *after* a successful database lookup — so the loaded entity is in scope. Logging the ID that came back from the database, rather than the `string id` off the URL, breaks the taint path at its source. It is also simply more correct: you log what you acted upon, not what was asked for.

This pattern is already established in the codebase — `ProjectController.cs:201` (soft-delete) does exactly this, which is why CodeQL does not flag it.

**This task changes no behaviour**: at each site the lookup has already succeeded, so `id` and the entity's ID are the same value.         public SecurityHeadersIntegrationTest(TiminuteApiFactory factory)

// and Blazor-framework middleware so it covers those responses too.
//
// X-Frame-Options is SAMEORIGIN, NOT DENY: Blazor WASM's OIDC stack renews the
// access token silently by loading the authorize endpoint in a hidden same-origin
// iframe. DENY blocks same-origin framing as well, which would break token renewal
// and log users out on expiry. SAMEORIGIN still stops cross-origin clickjacking.
//
// Content-Security-Policy is deliberately absent — it needs wasm-unsafe-eval for
// Blazor and its frame-src/frame-ancestors interact with that same silent-renew
// iframe. It needs its own testing pass; see the v2.3.1 spec.
app.Use(async (context, next) =>
{
    var headers = context.Response.Headers;
    headers["X-Content-Type-Options"] = "nosniff";
    headers["X-Frame-Options"] = "SAMEORIGIN";
    headers["Referrer-Policy"] = "strict-origin-when-cross-origin";
    await next();
});
```
