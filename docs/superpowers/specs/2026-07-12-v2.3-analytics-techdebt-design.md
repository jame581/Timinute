# v2.3 — Enhanced Analytics + Tech-Debt Sweep — Design

_Date: 2026-07-12. Status: approved in brainstorming session; implementation plan to follow._

## Goal

v2.3 pairs the Enhanced analytics P1 feature (new `/analytics` page + server-side aggregation, retrofitting the Dashboard off its load-all-tasks pattern) with the five tech-debt items scoped into v2.2 that did not ship.

## Scope

**Feature — Enhanced analytics:**
- Custom date range with presets driving all charts
- Server-side aggregation endpoints (per-day / per-project / per-tag / summary)
- Productivity trend chart vs `WorkdayHoursPerDay` target
- Per-tag breakdown (first analytics consumer of Tags, shipped v2.2)
- Dashboard retrofit to the new endpoints
- Client-side session caching (`AnalyticsService`)

**Tech debt (carried from v2.2):**
1. Build & Test workflow enable + green-run verification
2. Split `Server/Helpers/Constants.cs` per domain
3. Request/response logging middleware
4. API versioning (preparatory)
5. Server-side validation tests as integration tests

**Out of scope:** server-side caching (rejected — see Decisions), notifications, Pomodoro, external-ticket integrations, exports from the analytics page.

## Key decisions

| Decision | Choice | Why |
|----------|--------|-----|
| Analytics UI location | New `/analytics` page | Dashboard keeps its lightweight "today/this week" job; dedicated page owns range-driven exploration |
| Aggregation strategy | Slim SQL projection + in-memory group/sum on the server | `SUM` over `TimeSpan` doesn't translate to SQL (PR #45 lesson; `Duration` is a plain `TimeSpan`/`time` column). SQL does the user + range filter via `IX_TrackedTasks_UserId_StartDate` (shipped v2.2) and projects only needed columns; grouping + duration sums happen in server memory. No schema change. |
| Caching | Client-side memoization only | Per-user data changes constantly (running stopwatch); server-cache invalidation would need hooks in four controllers/services for a query that is millisecond-scale at self-hosted volumes. Client `AnalyticsService` caches per `(endpoint, from, to)` for the session — preset-hopping is instant — and is invalidated on local mutations. Follows the `UserProfileService` precedent (PR #44). |
| API versioning style | `Asp.Versioning.Mvc`, default 1.0, assume-default-when-unspecified | Existing routes unchanged; client+server ship as one unit so URL-segment migration is churn for nothing. Preparatory only. |

## Server design

### AnalyticsController

**Amendment (2026-07-12, from code study):** an `AnalyticsController` already exists (month-based endpoints `ProjectWorkTime`, `WorkTimePerMonths`, `AmountWorkTimeByMonth`, …). The four range endpoints are **added to it** — its constructor gains `ApplicationDbContext` (precedent: `TrackedTaskController`). The existing endpoints stay untouched.

`[Authorize]`, ownership-filtered on every action (repo invariant). Read-only `AsNoTracking` projection queries; the soft-delete global query filter applies automatically. The new endpoints carry `[ResponseCache(NoStore = true)]` so browser caching cannot serve stale data after the client invalidates its own cache (the controller-level `Default120` profile would otherwise apply).

All endpoints bind `[FromQuery] AnalyticsRangeDto` (`From`, `To`, `TzOffsetMinutes`):
- `From`/`To` UTC-normalized on arrival; `TzOffsetMinutes` (client's local UTC offset, validated ±14 h) shifts timestamps before **day grouping** so "today"/"per-day" match the user's local calendar day — the Dashboard's existing local-time attribution is preserved
- Validated via Data Annotations + `IValidatableObject`: non-default dates, `From <= To`, range ≤ 400 days → 422 via the global `InvalidModelStateResponseFactory` / problem+json path

| Endpoint | Returns |
|----------|---------|
| `GET /Analytics/summary?from&to` | Total tracked time, task count, active-day count, average per active day, target comparison (`WorkdayHoursPerDay × weekdays in range`; weekdays = Mon–Fri) |
| `GET /Analytics/daily?from&to` | One row per day with data: date, total duration, task count (zero-fill is client-side) |
| `GET /Analytics/projects?from&to` | Per project: id, name, color, total duration, task count |
| `GET /Analytics/tags?from&to` | Per tag: id, name, color, total duration, task count, plus a synthetic "untagged" bucket |

Query shape (all four): SQL filters `UserId == current && StartDate in range`, projects only `(StartDate, ProjectId/Name/Color, Tag fields, Duration)`, materializes, then groups and sums `Duration` in memory.

### DTOs

New `Shared/Dtos/Analytics/`: `AnalyticsSummaryDto`, `DailyAnalyticsDto`, `ProjectAnalyticsDto`, `TagAnalyticsDto`.

### Dashboard retrofit

Dashboard drops its "load all user tasks" call; consumes `summary` + `projects` + `daily` through the shared client `AnalyticsService`. Same visual output, small payloads.

## Client design

**Navigation:** "Analytics" item in the desktop sidebar; mobile gets it in the slide-up overflow sheet (bottom tabs unchanged). New `chart` icon case in `AuroraIcons.razor`.

**Page layout** (Aurora tokens only; no Radzen visual components):
1. **Header + range picker** — preset chips: This week, Last week, This month, Last 30 days, This year, Custom (two date inputs)
2. **Summary tile row** — total tracked, tasks, active days, avg/active day, vs-target (via cached `UserProfileService`)
3. **Daily trend chart** — hand-built SVG bars per day with a horizontal target line at `WorkdayHoursPerDay`; missing days render as zero; year-scale ranges bucket by week client-side
4. **Project breakdown** — donut + ranked list (reuses/extends the dashboard donut), project colors
5. **Tag breakdown** — ranked horizontal bars, tag colors, includes "untagged"

**AnalyticsService** (singleton, `UserProfileService` pattern): owns the four API calls; session cache keyed by the request URL (endpoint + from + to + tz). Shared by `/analytics` and Dashboard.

**Amendment (2026-07-12):** invalidation happens via a `DelegatingHandler` on the named API client — any successful non-GET response clears the analytics cache. One registration covers all ~13 existing mutation sites (and future ones) instead of sprinkling `Invalidate()` calls through every page.

**States:** skeleton tiles while loading; friendly empty state for ranges with no tasks.

## Tech-debt design

1. **Build & Test CI** — root cause was `disabled_manually` on GitHub; re-enabled 2026-07-12 (workflow id 20059071, now `active`). Verification: the v2.3 PR must show a green `Build & Test` run.
2. **Constants split** — **Amendment (2026-07-12):** `Server/Helpers/Constants.cs` turned out to be only 20 lines (Roles + Claims); the real debt is magic strings scattered in `Program.cs` (`"Timinute.ServerAPI"` ×4, authority fallback ×2, `"Default120"`). Scope: `partial class Constants` split into per-domain files (`Constants.Roles.cs`, `Constants.Claims.cs`, `Constants.Api.cs`) and those literals consolidated into `Constants.Api` / `Constants.CacheProfiles`; zero call-site churn for existing references.
3. **HTTP logging** — built-in `AddHttpLogging`/`UseHttpLogging`: method, path, status, duration; bodies and auth/cookie headers excluded. Off by default; `HttpLogging:Enabled` config flag (Docker env-var friendly). Standard pipeline placement; no user-id enrichment in v2.3.
4. **API versioning** — `Asp.Versioning.Mvc`, default `1.0`, `ReportApiVersions = true`, assume-default-when-unspecified; no route changes.
5. **Validation integration tests** — `WebApplicationFactory<Program>` class in `Server.Tests`: posts invalid DTOs through the real pipeline, asserts 422 `application/problem+json` (the `[ApiController]` short-circuit the ModelState-injection tests miss). Test auth scheme; DB swapped in the factory. Covers Analytics range validation too.

## Testing

- **Server:** controller tests on EF InMemory for logic/ownership; SQLite-backed tests proving the projection/filter queries translate (repo rule: InMemory hides translation failures); integration tests per above.
- **Client:** bUnit for range-picker preset logic and `AnalyticsService` cache/invalidation.
- **Reviews before commit:** `ef-repository-reviewer` (EF queries), `auth-config-reviewer` (Program.cs changes: logging, versioning).

## Release shape

One feature branch → one PR to `develop` (first green CI run), then release promotion per the usual flow. Spec (this doc) precedes the implementation plan per repo convention.

## Error handling summary

- Invalid range (from > to, > 400 days, non-UTC handled by normalization): 422 problem+json
- Unauthenticated: 401 via existing dual-scheme auth
- Empty ranges: 200 with empty collections / zeroed summary (client renders empty state)
