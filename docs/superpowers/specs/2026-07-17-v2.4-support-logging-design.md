# v2.4 — Support Logging (structured observability with Serilog)

_Date: 2026-07-17_
_Status: design approved, pending implementation plan_

## Purpose

Turn the app's existing `ILogger<T>` usage into production-grade, self-host-friendly
**observability logging** — structured, persistent, and correlatable — without rewriting
call sites. This is the first of two independent features in v2.4; the MCP server is the
second (`2026-07-17-v2.4-mcp-server-design.md`). The two ship in sequence: **Logging first**,
so there is good log signal in place before the MCP network surface is added.

The Duende → OpenIddict auth migration remains its own separate later release and is **out of
scope** here (see `timinute-auth-migration-decision.md`).

## Context

Current logging state (`Server/Program.cs`, `appsettings*.json`):

- Provider is the built-in `Microsoft.Extensions.Logging` with `builder.Logging.AddConsole()`.
- ~12 call sites already use `ILogger<T>` with **structured message templates**
  (e.g. `logger.LogInformation("Project with Id {ProjectId} soft-deleted along with {TaskCount} tasks.", …)`).
- Request-line logging exists via `AddHttpLogging` / `UseHttpLogging`, gated behind
  `HttpLogging__Enabled` (default off). Bodies and auth/cookie headers are deliberately never logged.
- `appsettings.json` `Logging:LogLevel` sets `Default: Information`, `Microsoft.AspNetCore: Warning`.

What is missing for real observability:

1. **Persistent / queryable sinks** — console-only logs vanish; self-hosters have nothing to grep.
2. **Structured (JSON) output** in production so `docker logs` and log scrapers can parse fields.
3. **Request correlation** — no id ties a request's log lines together.

## Decisions (from brainstorming)

- **Provider:** Serilog. Chosen over OpenTelemetry (needs a collector to add value — over-engineering
  for a personal self-hosted app) and over built-in JSON console (no file/UI out of the box).
- **Sinks:** Console (always on) + rolling File (opt-in). **No Seq** in v1.
- **Console format:** compact JSON when not `Development`; human-readable template in `Development`.
- **Correlation:** per-request correlation id enricher. **No user-id enricher** (privacy-minimal,
  consistent with the v2.3.1 log-forging hardening).
- **Call sites:** unchanged.
- **Severity ladder:** the app uses four levels — **DEBUG, INFORMATION (Info), WARNING, ERROR** — mapped
  to Serilog's `Debug` / `Information` / `Warning` / `Error`. Defaults: **`Debug`** minimum in
  `Development`, **`Information`** minimum in production (Debug excluded from prod to keep noise/volume
  down), both overridable per-namespace and via env var. Serilog's `Verbose` and `Fatal` are not used by
  app code. Level guidance below is written into the plan so call sites use them consistently.

## Scope

### 1. Add Serilog as the logging provider

Packages (pin exact versions in the implementation plan):

- `Serilog.AspNetCore`
- `Serilog.Sinks.File`
- `Serilog.Formatting.Compact` (for `CompactJsonFormatter`)

Wire via `builder.Host.UseSerilog((context, services, config) => config.ReadFrom.Configuration(context.Configuration)…)`
so levels/overrides live in `appsettings.json` and are env-var overridable for Docker. Remove
`builder.Logging.AddConsole()`.

`appsettings.json` gains a `Serilog` section seeded to preserve today's behaviour (production floor =
`Information`):

```json
"Serilog": {
  "MinimumLevel": {
    "Default": "Information",
    "Override": { "Microsoft.AspNetCore": "Warning" }
  },
  "File": {
    "Enabled": false,
    "Path": "/logs/timinute-.log",
    "RetainedFileCountLimit": 14
  }
}
```

`appsettings.Development.json` lowers the floor to `Debug`:

```json
"Serilog": { "MinimumLevel": { "Default": "Debug" } }
```

Both are env-overridable, e.g. `Serilog__MinimumLevel__Default=Warning`.

**Severity ladder — how the four levels are used** (documented in the plan; drives any new call sites):

| Level | When |
|---|---|
| `DEBUG` | Developer diagnostics: control-flow detail, values during a request. Dev floor; excluded in prod. |
| `INFORMATION` | Normal, notable events: request summary, a project/task created, an export produced. Prod floor. |
| `WARNING` | Handled-but-notable: duplicate-name 409, not-found on a delete, a revoked/expired token used. |
| `ERROR` | Unhandled/failed operations: exceptions, a tool call that failed, unexpected DB errors. |

The existing top-level `Logging` block is **removed** by the plan so Serilog is the single source of
truth for levels.

### 2. Console sink — always on

- **Development:** human-readable output template.
- **Non-Development:** `CompactJsonFormatter` → one JSON object per line to **stdout**, captured by
  Docker's `json-file` driver and surfaced by `docker logs` / `docker compose logs`. No volume, no
  extra config — this is the primary path for 99% of self-hosters.

### 3. Rolling File sink — opt-in

- Env-gated: `Serilog__File__Enabled=true`. Off by default (stock footprint unchanged).
- Daily rolling (`rollingInterval: Day`), path from `Serilog:File:Path` (default `/logs/timinute-.log`,
  a mountable Docker volume), `retainedFileCountLimit` from config (default 14), plus a per-file size cap
  set in the plan. File output uses the compact JSON formatter.

### 4. Correlation-id middleware + enricher

A small middleware, registered **early** in the pipeline (after `UseForwardedHeaders`, before request
logging):

- Reads an inbound `X-Correlation-Id` header if present and well-formed; otherwise generates one
  (`Guid.NewGuid().ToString("n")`).
- Pushes it as a `LogContext` property `CorrelationId` (`using (LogContext.PushProperty(...))`) so every
  log emitted during the request carries it.
- Echoes it back on the response as `X-Correlation-Id`.

Requires `.Enrich.FromLogContext()` in the Serilog config.

### 5. Replace HttpLogging with Serilog request logging

- Remove `AddHttpLogging(...)`, the `if (HttpLogging:Enabled) UseHttpLogging()` block, the
  `HttpLogging` appsettings block, and the `Microsoft.AspNetCore.HttpLogging` level override.
- Add `app.UseSerilogRequestLogging()` — one structured event per request (method, path, status,
  elapsed ms), automatically enriched with `CorrelationId` from the log context. Bodies/headers are
  still never logged (Serilog request logging logs the summary line only).

### 6. Docs

- `docs/DOCKER.md` gains a **"Logs"** section documenting **both paths**:
  - **Path 1 (Console → `docker logs`):** always on, JSON in prod; `docker logs -f timinute`,
    `docker compose logs -f timinute`; note that any Docker-log scraper (Loki, Fluent Bit, Datadog) can
    parse the JSON fields.
  - **Path 2 (File sink):** opt-in; the `Serilog__File__Enabled` + `Serilog__File__Path` env vars and the
    `volumes: - timinute-logs:/logs` compose snippet; note the container filesystem is ephemeral without a
    volume mount.
- `feature-roadmap.md` tech-debt: update the "Request/response logging middleware" row to record the
  Serilog replacement of `AddHttpLogging`.

## Non-goals

- No OpenTelemetry / OTLP export (can be added later as an extra Serilog sink).
- No Seq sink in v1.
- No user-id / PII enrichment.
- No change to the ~12 existing log call sites.
- No metrics or distributed tracing.

## Testing

- **Unit:** correlation-id middleware — generates when the header is absent, honours a valid inbound
  `X-Correlation-Id`, and echoes `X-Correlation-Id` on the response.
- **Regression:** the existing `WebApplicationFactory<Program>` integration suite must pass unchanged with
  Serilog swapped in (proves boot + pipeline still work).
- **`/verify` (skill):** launch the app, confirm structured logs emit to console; enable the file sink via
  env and confirm a rolling file is written.

## Risks

- **Two level sources:** leaving both `Logging` and `Serilog` sections would be confusing — the plan
  removes the `Logging` block so Serilog is the single source of truth.
- **File path on Windows dev vs Linux container:** default `/logs/...` is Linux-oriented; on local Windows
  dev the file sink stays **off by default**, so no dev-time path breakage. Documented.

## Ordering

Ships **before** the MCP server in v2.4, so request correlation + structured logs are already in place when
the `/mcp` surface is added.
