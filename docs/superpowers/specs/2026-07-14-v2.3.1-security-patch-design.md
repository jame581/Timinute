# v2.3.1 — Security patch (code scanning cleanup)

_Date: 2026-07-14_
_Status: design approved, pending implementation plan_

## Purpose

Close the 12 open GitHub code-scanning alerts on `jame581/Timinute`, and add the
dependency-update automation whose absence caused several of them. This is a
**patch release**: no new features, no auth changes, no behavioural change for users.

The Duende IdentityServer replacement — originally scoped as v2.4 — is explicitly
**out of scope here** and is deferred to its own release. See "Deferred to v2.4" below,
which records the research so that release starts from a decision rather than a blank page.

## Context

Code scanning runs via GitHub's **default setup** (there is no `codeql.yml` in
`.github/workflows/`). The 12 open alerts fall into four groups, and — importantly —
most have a fix that is *better code*, not a suppression.

| # | Severity | Rule | Location |
|---|----------|------|----------|
| 23 | high | `cs/user-controlled-bypass` | `Areas/Identity/Pages/Account/ExternalLogin.cshtml.cs:98` |
| 24 | high | `cs/web/xss` | `Areas/Identity/Pages/Account/ExternalLogin.cshtml:19` |

## Scope

### 1. Delete the dead ExternalLogin pages — closes #23, #24

`ExternalLogin.cshtml` and `ExternalLogin.cshtml.cs` are scaffolded Identity leftovers.
**No external authentication provider is registered anywhere in the application.**
`Server/Program.cs:54` carries an explicit comment confirming this is deliberate — the
`ApplicationDefinedPolicy` scheme is registered with `displayName: null` precisely so the
Identity UI does not render "or continue with" provider buttons.

The pages are therefore unreachable. Deleting them closes both alerts and removes real
attack surface rather than masking it.

**Implementation note:** check for and remove now-dangling references — the external-login
section of `Login.cshtml`, and `Areas/Identity/Pages/Account/Manage/ExternalLogins.cshtml`
if present. The build must stay green.

### 2. Log the entity ID, not the route parameter — closes #18–#22

Every flagged log call sits *after* a successful database lookup, so the loaded entity is
already in scope. Logging the value that came back from the database — rather than the raw
`string id` off the URL — breaks the taint path at its source.

```csharp
// before — TrackedTaskController.cs:189
logger.LogInformation("Tracked task with Id {TaskId} was soft-deleted.", id);

// after
logger.LogInformation("Tracked task with Id {TaskId} was soft-deleted.", trackedTaskToDelete.TaskId);
```

This is not merely alert-appeasement: it is more correct, because it logs what was actually
acted upon rather than what was requested. No API signatures change.

Five call sites:
- `TrackedTaskController.cs:189` (soft-delete), `:213` (restore), `:263` (purge)
- `ProjectController.cs:234` (restore), `:284` (purge)

### 3. Validate `ReturnUrl` on GET — closes #25, #26

These two are **false positives for XSS** — Razor encodes the attribute value, so nothing
executes. They are being fixed anyway because they point at something real.

`LoginWith2fa.cshtml.cs:82` assigns `ReturnUrl = returnUrl` straight from the query string.
The value is only ever validated at POST time, by `LocalRedirect(returnUrl)` on line 112.
Validating on GET with `Url.IsLocalUrl()` and falling back to `~/` is proper defence-in-depth
against open redirect, and it breaks the taint path CodeQL is reporting.

Apply the same treatment in `Login.cshtml.cs` (`:81`, `:95`), which has the identical pattern.

### 4. Refresh the vendored client libraries — targets #3–#7

The libraries under `wwwroot/lib` are vendored with **no `libman.json` and no `package.json`**,
so nothing has ever updated them. They are badly stale:

| Library | Vendored | Current |
|---------|----------|---------|
| jquery-validation | **1.17.0** (2017) | 1.21.x |
| jquery | 3.5.1 (2020) | 3.7.x |
| jquery-validation-unobtrusive | 3.2.11 | 4.x |

Refresh them. Introduce a `libman.json` so the versions are declared and updatable rather
than frozen by a copy-paste.

**Honest expectation, agreed with the user:** the `js/redos` findings are in
jquery-validation's URL-validation regex, which is upstream third-party code. **Refreshing
the library may not close all five alerts.** We are explicitly *not* suppressing them —
no `paths-ignore`, no dismissals. If alerts survive the bump, they stay open and visible.
A red dashboard we understand is better than a green one we engineered.

Only `_ValidationScriptsPartial.cshtml` consumes these libraries (the Identity Razor Pages);
the Blazor client does not. Regression surface is limited to client-side validation on the
Identity forms — server-side validation is unaffected and remains authoritative.

### 5. Add `dependabot.yml`

There is no `.github/dependabot.yml` today. This is the root cause of item 4: a 2017-era
library sat un-upgraded for nine years.

Enable it for `nuget` and `github-actions`. (The vendored `wwwroot/lib` assets are not
package-managed, hence the `libman.json` in item 4 — without it, Dependabot cannot see them.)

### 6. Safe security headers

Only HSTS is set today (`Program.cs:206`). Add the low-risk, no-behavioural-change headers:

- `X-Content-Type-Options: nosniff`
- `X-Frame-Options: DENY`
- `Referrer-Policy: strict-origin-when-cross-origin`

**Content-Security-Policy is deliberately excluded from this patch.** A correct CSP for
Blazor WebAssembly requires `wasm-unsafe-eval` and must be validated against the OIDC
login-redirect flow. Getting it subtly wrong breaks the app load or the login path — not
something a patch release should risk. It is deferred.

## Explicitly out of scope

- **Any Duende IdentityServer change.** Staying on Duende for v2.3.1.
- **Content-Security-Policy.** Deferred — needs real testing against Blazor WASM + OIDC.
- **Suppressing or dismissing any alert.** If a fix does not close an alert, it stays open.

## Deferred to v2.4 — Duende replacement

Research completed 2026-07-14. Recorded here so v2.4 starts from a decision.

**Why leave Duende at all?** Not cost. Duende's [Community Edition](https://duendesoftware.com/products/communityedition)
is free (under $1M revenue; personal projects need no licence at all), and v8 runs in trial
mode with no request limit and no shutdown. The real reason is **downstream freedom**: Duende
is source-available, not OSI open source — dual-licensed RPL + commercial since Nov 2022. Anyone
who forks Timinute and deploys it *commercially* inherits a licence obligation. Timinute's own
maintainer is **not** liable for that (the licence attaches to whoever deploys to production),
but shipping an open-source project whose auth dependency is not freely usable in production
contradicts the project's goal.

**Decision: OpenIddict.** Migration surface is small — Duende is confined to two
`PackageReference` lines and the `IdentitySetup()` function in `Server/Program.cs` (~65 lines).
The Blazor client is provider-agnostic (`AddOidcAuthentication` + `RemoteAuthenticatorView`)
and barely changes.

| | Open.IdentityServer | **OpenIddict** | Keycloak |
|---|---|---|---|
| Licence | Apache 2.0 | Apache 2.0 | Apache 2.0 |
| Age | 8 weeks (repo created 2026-05-18) | 11 years | 12+ years |
| Downloads (current version) | 723 | 635,000 | n/a (CNCF) |
| Maintainers | 2, one company | 1 primary + ABP/Orchard/Umbraco depend on it | Red Hat + CNCF |
| Security advisories filed | **zero, ever** | zero CVEs in 11 years | active programme |
| Deployment | in-process .NET | in-process .NET | **separate JVM container** |
| Migration cost | ~65 lines | **1–2 days** | weeks + permanent Java toolchain |

**Why not Open.IdentityServer**, despite being the cheap swap:
- Its free tier is a funnel to paid components, and **automatic key management — the exact
  Duende feature Timinute relies on today (`Program.cs:319-326`) — is on their roadmap as a
  *paid component*, targeted Sept 2026.** Leaving one paywall to walk into another.
- Rock Solid Knowledge was acquired by Jonas Software (Constellation Software) in Dec 2024.
- `PublicOrigin` was removed (issue #46) — Timinute runs behind a reverse proxy with forwarded
  headers, so this bites directly.
- Implicit/hybrid/ROPC flows remain **enabled by default** for IdentityServer4 migration
  compatibility.
- v1.0.1 shipped with a null-handling bug in `DefaultKeyMaterialService` (issue #31, fixed
  after release) — the key-material path is its least-exercised code.

**Why not Keycloak:** a separate Quarkus/JVM container (~1.25 GB RAM per their own sizing),
its own database, a FreeMarker theme rewrite of the 41 Identity Razor Pages, and — the blocker —
ASP.NET Core Identity's V3 password hashes (256-bit subkey, HMACSHA512) **cannot be verified by
Keycloak's stock `pbkdf2-sha512` provider**, which hardcodes a 512-bit derived key size. Migrating
passwords would require shipping a custom Java SPI plugin from a C#-only project, forever. It would
destroy the one-container self-host story.

**Known costs of the OpenIddict path** (to plan for in v2.4):
- OpenIddict is a protocol *framework*, not a turnkey server: **you write the authorization
  controller** (~386 LOC in the official [Balosar sample](https://github.com/openiddict/openiddict-samples/tree/dev/samples/Balosar),
  which is exactly Timinute's shape — hosted Blazor WASM + ASP.NET Core Identity + PKCE public client).
- No `AddInMemoryClients` equivalent — client storage is EF-backed, adding four tables plus a
  startup seeder. One migration.
- **No automatic key management** (same as Open.IdentityServer — this is a wash between them).
  Requires *two* certificates: signing and encryption. Auth codes and refresh tokens are always
  encrypted even if access-token encryption is disabled.
- Razor Pages Identity UI, `ApplicationUser`, `AppSingInManager`, and
  `ApplicationUserClaimsPrincipalFactory` all stay **untouched**.
- Support is community-only: GitHub issues from non-sponsors are auto-closed.

Also update `Client/Components/Landing/LandingFeatures.razor:8`, which name-drops Duende.
