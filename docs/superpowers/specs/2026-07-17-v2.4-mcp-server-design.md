# v2.4 — MCP Server (AI access to a user's time data)

_Date: 2026-07-17_
_Status: design approved, pending implementation plan_

## Purpose

Expose Timinute's time-tracking data to AI assistants (Claude Desktop / Claude Code, and any
MCP-compatible client) by hosting a **Model Context Protocol server inside the app**. A user
creates a Personal Access Token in Settings, pastes it into their MCP client, and the assistant can
then query and mutate **that user's** projects, time entries, and analytics on their behalf
("log 45 minutes on Project X", "what did I track last week?").

Second of two independent v2.4 features; ships **after** Support Logging
(`2026-07-17-v2.4-support-logging-design.md`).

## Key design decision — PAT auth, not OAuth (why these features are decoupled)

The MCP spec models a remote (HTTP) server as an OAuth 2.0 *Resource Server*, and full compliance
would pull in Dynamic Client Registration, protected-resource metadata, and PKCE — all of which live
in the Authorization Server (Duende today, OpenIddict later). Going that route would **tightly couple
MCP to the auth provider** and force the OpenIddict migration to land first.

For a self-hostable, effectively single-user-per-instance personal time tracker, full OAuth DCR is
**over-engineering (YAGNI)**. Developers already wire MCP servers with a **bearer token in the client
config**. So:

> **MCP v1 authenticates with a user-generated Personal Access Token (PAT)** — a scoped bearer
> credential the user creates in Settings and pastes into their MCP client, validated as a bearer
> token by the API.

A PAT is **provider-agnostic**: it does not care whether Duende or OpenIddict issues the app's login
tokens. This fully **decouples MCP from the auth migration**, which stays its own later release.

## Context (reuse surface)

- **Auth pipeline** (`Server/Program.cs:41-66`): a default `ApplicationDefinedPolicy` **policy scheme**
  already inspects the `Authorization` header — `Bearer …` → `JwtBearer`, otherwise → Identity cookie.
  This is the exact insertion point for a third branch.
- **Controllers** are `[Authorize]` and resolve the user via `User.FindFirstValue(Constants.Claims.UserId)`,
  then scope every query with a `UserId ==` predicate (e.g. `TrackedTaskController.cs:47,54-57`).
- **Data access:** `IRepositoryFactory` → `IRepository<T>` (`GetPaged`, ownership filters, includes) +
  AutoMapper (`MappingProfile`). Soft-delete global query filter is already in place.
- **SDK:** `ModelContextProtocol.AspNetCore` — `AddMcpServer().WithHttpTransport().WithToolsFromAssembly()`,
  `MapMcp("/mcp")`, Streamable HTTP. Tools are `[McpServerToolType]` classes with `[McpServerTool]` +
  `[Description]` methods. Microsoft guidance: **do not duplicate controller logic in tools — call a shared
  service**.

## Decisions (from brainstorming)

- **Tool surface:** read **and** write (query + create project + log/edit/delete time entries).
- **Token scopes:** per-token `read` or `read_write`; read tokens cannot call mutating tools.
- **Shared logic:** **extract a thin app-service layer** (option A) for the MCP-exposed operations;
  the relevant controller actions delegate to the same services (single source of truth for
  validation/ownership).
- **Enable flag:** `Mcp__Enabled`, default **true** (inert without a valid PAT).

## Scope

### 1. PAT persistence

New EF entity **`PersonalAccessToken`**:

| Field | Notes |
|---|---|
| `Id` | PK |
| `UserId` | FK → `ApplicationUser` (cascade delete with the user) |
| `Name` | user-supplied label |
| `TokenHash` | **SHA-256** of the full token. Tokens are high-entropy random, so a fast hash is correct — **not** a password hasher. |
| `Prefix` | first ~8 chars stored in plaintext for list display + fast candidate lookup |
| `Scopes` | enum/string: `read` or `read_write` |
| `CreatedAt` / `LastUsedAt` | `DateTimeOffset` (UTC, per house style) |
| `ExpiresAt?` | optional expiry |
| `RevokedAt?` | soft revoke |

New migration `AddPersonalAccessTokens`. Index on `Prefix` (lookup) and `UserId` (listing).

**Token format:** `tmn_pat_<base64url of 32 random bytes>`. Generated with a CSPRNG. Shown **once** at
creation and never retrievable again.

### 2. PAT authentication scheme

New **`PatAuthenticationHandler`** (scheme name `"Pat"`):

1. Read `Authorization: Bearer tmn_pat_…`.
2. Extract prefix → look up candidate token(s).
3. Constant-time compare the SHA-256 hash; reject on mismatch.
4. Reject if `RevokedAt != null` or `ExpiresAt` is past.
5. Stamp `LastUsedAt` (best-effort, must not block/fail the request).
6. Build a `ClaimsPrincipal` carrying the **same `UserId` claim** controllers already read, plus a
   `scope` claim (`read` / `read_write`).

**Wiring:** extend the `ApplicationDefinedPolicy` `ForwardDefaultSelector` — if the bearer value starts
with `tmn_pat_` → forward to `"Pat"`; else → `JwtBearer`. **No change to existing JWT/cookie behaviour.**

### 3. Token management API + UI

- **`PatController`** — `[Authorize]` under the **existing** cookie/JWT schemes (explicitly **not**
  PAT-authenticated, so a token can never mint or list tokens):
  - `GET /Pat` → list (metadata only: name, prefix, scope, created, last-used, expiry — never the secret).
  - `POST /Pat` → create; returns the plaintext token **once**.
  - `DELETE /Pat/{id}` → revoke (ownership-checked).
- **Settings page `/settings/tokens`** (Aurora-styled, following the Profile/Preferences patterns):
  list tokens, "Create token" flow with **copy-once** reveal + scope choice, revoke-with-confirm, and a
  short "Connect your AI client" help block (the `/mcp` URL and how to paste the token).

### 4. App-service layer (shared logic)

Extract thin, `userId`-parameterised services encapsulating ownership + validation for the MCP-exposed
operations, and refactor the corresponding controller actions to delegate:

- `ProjectAppService` — list, create (dup-name 409 rule).
- `TimeEntryAppService` — search, log (create), update, delete (reuses the `ProjectId` ownership /
  whitespace-normalisation hardening from v2.3).
- `AnalyticsAppService` — range summary.

Only the ~4-6 actions MCP needs are refactored — not a full 6-controller rewrite.

### 5. MCP tools

`[McpServerToolType]` classes; each `[McpServerTool]` method has a clear `[Description]` and operates
**only** on the PAT principal's `UserId`, delegating to the app-services:

| Tool | Scope | Delegates to |
|---|---|---|
| `list_projects` | read | `ProjectAppService.List` |
| `search_time_entries` (from/to, project, tag, name) | read | `TimeEntryAppService.Search` |
| `get_analytics_summary` (range) | read | `AnalyticsAppService.Summary` |
| `create_project` (name, color) | read_write | `ProjectAppService.Create` |
| `log_time` (project, start, end/duration, description, tags) | read_write | `TimeEntryAppService.Log` |
| `update_time_entry` (id, fields) | read_write | `TimeEntryAppService.Update` |
| `delete_time_entry` (id) | read_write | `TimeEntryAppService.Delete` |

**Scope gating:** write tools require the `read_write` scope claim; a `read` token invoking them gets an
authorization error. Enforced centrally (authorization filter/helper), not per-tool ad-hoc.

**Live stopwatch is out of MCP:** the running timer is client-side session-storage state, not server
state. MCP logs **completed** entries and queries data; it does not start/stop the live timer.

### 6. Endpoint & hosting

- `AddMcpServer().WithHttpTransport().WithToolsFromAssembly()`; `MapMcp("/mcp").RequireAuthorization(...)`
  under the `"Pat"` scheme.
- Gated by `Mcp__Enabled` (default true).
- CORS: MCP desktop/CLI clients connect **server-to-server**, not from a browser, so no broad CORS is
  opened. If a browser-based MCP tester is ever needed it can be scoped to `/mcp` behind a flag; not in v1.
- All MCP code lives in an isolated `Mcp/` folder so the **prerelease** SDK dependency is easy to pin,
  bump, or remove.

## Non-goals

- No OAuth 2.0 / Dynamic Client Registration / protected-resource metadata (deliberate — see key decision).
- No MCP **resources** or **prompts** in v1 — tools only.
- No driving of the live client-side stopwatch.
- No admin/cross-user access; every tool is single-user scoped by the PAT.
- No dependency on the OpenIddict migration.

## Testing

- **Unit:** `PatAuthenticationHandler` (valid / revoked / expired / wrong-prefix / hash-mismatch /
  scope-gating), token generation + hashing (format, uniqueness, constant-time compare).
- **Integration (`WebApplicationFactory`):** `/mcp` rejects requests with no token and with a `read` token
  calling a write tool; a read tool returns **only** the caller's data (ownership isolation across two
  seeded users); `PatController` never returns a secret and enforces ownership on revoke.
- **App-service parity:** tests proving a controller action and its MCP tool produce the same result via
  the shared service (no logic drift).
- **`/verify` (skill):** create a token in the UI, connect a real MCP client, `list_projects`, `log_time`,
  confirm the entry appears in the app; confirm a read-only token is refused a write.

## Risks

- **Prerelease SDK:** `ModelContextProtocol.AspNetCore` is prerelease. Mitigated by exact version pinning,
  code isolation under `Mcp/`, and the `Mcp__Enabled` off-switch.
- **Token leakage:** plaintext shown once, only a SHA-256 hash + prefix stored; revoke available; optional
  expiry. Tokens never logged (correlation-id logging from part 1 does not capture auth headers).
- **Scope-gating gap:** must be enforced centrally, or a write tool could slip past a read token —
  covered by the scope-gating tests above.
- **Self-host exposure:** `/mcp` is a new network surface. It is auth-required (no PAT = no access) and
  flag-gated; documented in `docs/MCP.md` and DOCKER.md.
